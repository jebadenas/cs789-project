#### What is the Force-Layout Graph?

The force-layout graph is a **directed weighted graph** that visualises peer-scoring relationships within a single team. It uses the **Fruchterman–Reingold spring layout algorithm** — a physics simulation where each edge acts like a spring. Higher scores create stronger springs, so students who rate each other highly are pulled closer together, while weakly-connected students drift apart.

**What it shows:**
- Each **node** is a student. Each **directed edge** represents a score one student gave to another.
- The graph is **model-independent** — it visualises the raw score matrix directly, not any IWF model output.

**Visual encodings:**
| Element | What it encodes |
|---------|----------------|
| Node size | Weighted in-degree — total score received from all peers |
| Node colour | RdYlGn gradient based on in-degree (green = received more, red = received less) |
| Edge thickness | Score magnitude (thicker = higher score, scaled 1–8px) |
| Edge colour | RdYlGn gradient (green = high score, red = low score) |
| Arrow direction | Triangle at 75% along edge pointing toward the recipient |
| Spatial position | Students who score each other highly cluster together |

**What's excluded:**
- **Self-scores** are excluded from edges — the graph only shows inter-student relationships. This means out-degree (total points given as edges) can vary between students, because each student allocates a different portion of their fixed budget (60 pts) to themselves. A lower out-degree signals a higher self-score.
- **Non-submitters** have no outgoing edges (NaN column) but still appear as nodes receiving incoming edges.

**How to read it:**
- Big green nodes = well-rated students (high scores received)
- Small red nodes = poorly-rated students or free-riders
- Thick green edges = strong positive ratings between students
- Clusters of tightly-connected nodes = sub-groups who rate each other highly
- Isolated nodes = students disconnected from the scoring consensus

### 4. Grading Engine — How each model works (quick reference)

| Model | Algorithm | Key insight |
|-------|-----------|-------------|
| **Baseline** | Mean of peer scores (exclude self, NaN-aware) | Simple, no weighting |
| **WebPA** | (Scores received / team average) × 10. Includes self-scores | "Vote share" approach |
| **PeerRank** | Iterative: normalise columns → credibility-weight → fixed-point (α=0.1) | Low-rated students' votes dampened |
| **PeerHITS** | authority = M × hub; hub = M^T × authority; L2-normalise each step | Dual score: contribution quality vs. rating quality |
| **-Impute** | Replace NaN columns with uniform scores before running | Non-submitter → neutral (10.0) |
| **-Exclude** | Remove non-submitter rows/cols, run on submatrix, reinsert as NaN | Non-submitter → N/A |

### 5. Case Study: PeerHITS Amplification (Team 16, S1 2023 Session 3, showcase poster)

**Score matrix:**

```
              Wei    Ma    Xie    Yan    Wang
Wei          12.0   0.0   12.0   0.0   12.0
Ma           12.0   0.0   12.0   0.0   12.0
Xie          12.0   0.0   12.0   0.0   12.0
Yan          12.0   0.0   12.0   0.0   12.0
Wang         12.0  60.0   12.0  60.0   12.0
```

Ma and Yan gave **all 60 points to Wang** (0 to everyone else, 0 to themselves).

**IWF comparison:**

| Student | Baseline | PeerRank | WebPA | PeerHITS |
|---------|----------|----------|-------|----------|
| Wei | 6.000 | 5.906 | 6.000 | 1.054 |
| Ma | 9.000 | 7.326 | 6.000 | 1.991 |
| Xie | 6.000 | 5.906 | 6.000 | 1.054 |
| Yan | 9.000 | 7.326 | 6.000 | 1.991 |
| **Wang** | **36.000** | **23.536** | **26.000** | **43.911** |

**Why PeerHITS amplifies Wang beyond all other models:**

PeerHITS produces two scores: **authority** (contribution quality → IWF) and **hub** (rating quality). These reinforce each other in a feedback loop:

1. Ma & Yan gave Wang 60 pts each → Wang's authority shoots up
2. High authority for Wang → Ma & Yan get high hub scores (20.2) for "correctly" identifying the top contributor
3. High hub scores mean Ma & Yan's ratings carry *more weight* next iteration → Wang's authority goes even higher
4. Wei & Xie rated everyone equally (12 each) → low hub scores (4.5) — their flat ratings don't align with the emerging consensus

This is a **reinforcement spiral** inherent to HITS-style algorithms. PeerRank dampens this via its learning rate (α=0.1), landing at 23.5. PeerHITS has no dampening — pure matrix multiplication amplifies the effect to 43.9.

**Takeaway:** PeerHITS is more sensitive to extreme/concentrated scoring patterns. This is both a strength (detects dominant contributors) and a risk (amplifies potential manipulation).

### 6. Team Dynamics Classification (RQ3 discussion)

#### Background

Classical clustering algorithms (Louvain, Leiden, Spectral) are unsuitable for intra-team analysis — teams of 5–8 students are far below the scale where modularity optimisation produces meaningful partitions. The recommended approach computes graph metrics directly from each `ScoreMatrix`, then applies threshold-based classification rules. This is the method used by the **CATME** system (Ohland et al., 2012), deployed in engineering capstone courses worldwide for marking-pattern classification.

Full research: `docs/team-dynamics-research.md`

#### Proposed metrics (computed per team per matrix)

| Metric | What it detects |
|--------|----------------|
| Reciprocity | Symmetric vs. asymmetric scoring relationships |
| Gini coefficient | Score concentration — free-riders or dominant members |
| Score variance per rater | Differentiating raters vs. flat/strategic raters |
| Asymmetry | One-directional imbalances (conflict signal) |
| Clustering coefficient | Tight sub-group mutual inflation (collusion signal) |
| Degree assortativity | Whether high-rated students preferentially rate each other |

#### Classification labels

| Label | Key signal |
|-------|-----------|
| **Cohesive** | Moderate reciprocity, low asymmetry, low Gini |
| **Collusive** | Low variance, uniformly high scores, high clustering |
| **Free-rider present** | High Gini, one student's in-degree well below team mean |
| **Dominant member** | High Gini, one node with much higher in-degree centrality |
| **Conflict** | High asymmetry, low reciprocity |

Thresholds calibrated empirically at dataset percentiles (e.g., "high Gini" = above 75th percentile across all 136 matrices).

#### Evaluation: two lenses

**Lens 1 — Synthetic ground truth (extends Phase 4 attack simulation):**
Generate teams with known true contribution vectors and specific injected dynamics. Measure RMSE of each model's output per dynamic type. Directly answers: *which model is most accurate under each team condition?*

**Lens 2 — Real data (Delta as proxy):**
Classify real teams by label, then stratify cross-model divergence (Delta) by label. Hypothesis: Cohesive teams → low Delta (models agree); Collusive/Dominant-member teams → high Delta (models react differently to unusual patterns). Validates both the labels and the Delta signal as a reliable red flag — the core claim of RQ3.
